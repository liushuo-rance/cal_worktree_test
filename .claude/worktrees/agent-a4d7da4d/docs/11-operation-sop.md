# 加班记录分析系统 - 操作工作流SOP

> 本文档定义系统日常运营的标准操作流程(Standard Operating Procedures)，确保数据处理的一致性和合规性。

---

## ⚠️ 重要声明：关于"累计"值的处理

### 背景说明

员工在Markdown记录文件中通常包含"累计X小时"的声明，例如：
```
2025.09.10，晚上3.5小时，累计3.5小时
2025.9.18，晚上4.5小时，累计19小时。
```

### 关键问题

**系统在处理时，完全不参考文件中声明的"累计"值。**

原因如下：
1. **未区分加班类型**：员工记录未区分工作日延时、周末、法定节假日，混为一谈进行累加
2. **不符合合规要求**：《劳动法》对不同加班类型有差异化规定（1.5x/2.0x/3.0x，可调休/不可调休）
3. **可能包含非加班记录**：如"午餐会"、"学习奖励"等不应计入加班的条目

### 系统的正确做法

1. **独立计算**：系统根据日期自动判定加班类型，按《劳动法》规则独立计算
2. **仅提取展示**：文件中的"累计X小时"仅作为文本提取，在界面展示供参考
3. **不参与校验**：不进行"计算累计 vs 声明累计"的校验，不因此阻断保存流程

### 对操作的影响

- 导入时无需关注"累计值校验失败"的提示
- 员工沟通时以系统计算的合规余额为准
- 如员工对余额有疑问，解释系统按《劳动法》规则独立计算

# SOP-001: Markdown文件导入与批量处理

## 1. 流程概述

将员工以Markdown格式记录的加班数据导入系统，进行解析、标准化和持久化存储。

### 输入
- 员工提交的 `.md` 格式加班记录文件
- 文件位于 `employee_ot_record/` 目录

### 输出
- 解析后的结构化数据存储到SQLite数据库（分表存储）
  - `overtime_records` 表：存储加班记录（时间存正数，区分工作日早晨/午休/晚间、周末、法定假日）
  - `comp_off_balances` 表：存储周末加班产生的可调休余额
  - `leave_records` 表：存储请假记录（可选）
- 导入报告（成功/失败统计，按类型分别统计）
- 解析错误日志（如有）

### 数据存储架构

系统采用**表分离**设计，确保加班和调休分别管理：

| 表名 | 存储内容 | 统计方式 |
|------|----------|----------|
| `overtime_records` | 加班记录 | 按 `overtime_type` 分组统计各类型加班时长（时间存正数） |
| `comp_off_balances` | 调休余额 | 按 `acquired_date` 和 `status` 统计可用余额（时间存正数） |
| `leave_records` | 请假记录 | 按请假类型和时长统计（时间存正数） |
| `comp_off_usage_records` | 调休使用 | 记录调休使用明细（时间存正数） |

**重要说明**：
- 所有时间字段均存储**正数**，不存储负值
- 工作日延时加班（早晨、午休、晚间）存储在 `overtime_records`，**不产生**调休余额
- 周末加班存储在 `overtime_records`，同时在 `comp_off_balances` 生成可调休余额
- 调休使用时，在 `comp_off_usage_records` 记录使用明细，并从 `comp_off_balances` 按FIFO原则抵扣
- 累计结果通过**业务逻辑代码**计算，不直接操作数据库字段进行加减

# SOP-003: 逐行审批模式操作

## 1. 流程概述

对低置信度或格式不规范的记录进行人工审查和确认，确保数据准确性。

### 类型分离处理原则

逐条审批时，系统根据记录类型**分别处理**：

| 记录类型 | 存储目标表 | 处理逻辑 |
|----------|------------|----------|
| **加班(OVERTIME)** | `overtime_records` | 细分为 weekday_morning/lunch/evening/weekend/holiday，时间存正数 |
| **调休(COMP_OFF)** | `comp_off_balances` + `comp_off_usage_records` | 抵扣周末加班余额，记录使用明细，时间存正数 |
| **请假(LEAVE)** | `leave_records` | 根据请假类型（年假/事假/病假）分别统计 |

### 工作时间定义（用于判定加班时段）

- 工作日：周一到周五
- 上午：08:30 - 12:00
- 午休：12:00 - 13:00
- 下午：13:00 - 17:30
- 工作日延时加班：08:30前、12:00-13:00、17:30后

---

## 2. 操作步骤

### 步骤1: 启动逐行审批模式

```bash
# 对指定文件启动审批模式

# 只审批特定行
python -m ot_system review --file employee_ot_record/lijunjie.md --lines 3,5,8
```

# SOP-006: 年度节假日维护

## 1. 流程概述

每年11-12月，从国务院官网获取下一年度节假日安排通知，导入系统以确保加班计算符合实际法定节假日安排。

### 前置条件
- 国务院已发布下一年度节假日安排通知（通常在11月中下旬）
- 具有系统管理员或HR管理员权限

### 输出
- 下一年度的 `national_holidays` 表数据
- 下一年度的 `adjusted_workdays` 表数据

---

## 2. 操作步骤

### 步骤1: 获取国务院通知

1. 访问国务院官网: http://www.gov.cn/zhengce/zhengceku/
2. 搜索关键词: "国务院办公厅关于20XX年部分节假日安排的通知"
3. 打开通知页面，复制正文部分

**通知文本示例**:
```
国务院办公厅关于2026年部分节假日安排的通知发布：
一、元旦：1月1日（周四）至3日（周六）放假调休，共3天。1月4日（周日）上班。
二、春节：2月15日（农历腊月二十八、周日）至23日（农历正月初七、周一）放假调休，共9天。
...
```

### 步骤2: 打开节假日管理功能

**GUI方式**:
```
系统管理 → 节假日管理 → 从文本导入
选择年度: [2026 ▼]
```

**CLI方式**:
```bash
python -m ot_system holidays import --year 2026 --interactive
```

### 步骤3: 粘贴通知文本

1. 将复制的通知文本粘贴到文本框
2. 检查文本完整性（应包含"一、"到"七、"或"八、"的所有节日）
3. 点击"开始解析"

### 步骤4: 确认解析结果

**核对清单**:
- [ ] 元旦放假日期正确（通常为3天）
- [ ] 春节放假日期正确（关注农历日期是否正确识别）
- [ ] 清明节是否为4月4日或5日
- [ ] 劳动节是否为5月1日开始
- [ ] 端午节农历五月初五是否正确识别
- [ ] 中秋节农历八月十五是否正确识别
- [ ] 国庆节是否为10月1-7日（或通知的具体安排）
- [ ] 所有调休上班日是否已列出

**常见问题处理**:

| 问题 | 处理方式 |
|------|----------|
| 农历日期识别错误 | 手动选择正确的公历日期 |
| 调休上班日缺失 | 手动添加缺失的日期 |

## 3. 异常处理

### 3.1 解析失败

**症状**: 点击"开始解析"后提示"未能识别任何节假日"

**处理**:
1. 检查文本是否包含节数编号（"一、"、"二、"等）
2. 检查是否包含"放假"关键词
3. 尝试手动复制，避免复制网页格式
4. 如仍失败，使用"手动添加"方式逐条录入

### 3.2 农历日期识别错误

**症状**: 春节日期与日历不符

**处理**:
1. 查阅万年历确认正确的公历日期
2. 在确认界面手动修正日期
3. 标记为"已人工核对"

### 3.3 数据已存在

**症状**: 提示"2026年节假日数据已存在"

**处理**:
- 如数据正确：选择"保留现有数据"
- 如数据有误：选择"覆盖更新"
- 如需对比：选择"逐条对比"
