# 加班记录分析系统 - TDD阶段规划文档

## 概述

本文档定义加班记录分析系统的分阶段TDD开发计划，共6个阶段，每个阶段有明确的目标、测试策略和验收标准。

---

## 阶段1：基础架构与数据模型

### 目标
建立系统基础架构，包括时间处理工具和数据库Schema。

### 交付模块
1. **时间工具** (`src/utils/time_utils.py`)
   - 时分与总分钟数互转
   - 工作时间判断 (08:30-12:00, 13:00-17:30)
   - 加班时段分类 (早晨/午休/晚间)
   - 跨时段加班计算

2. **数据库Schema** (`src/db/schema.py`)
   - 8张核心表设计
   - 约束：正数时间、枚举类型、外键
   - 索引优化
   - 统计视图

### 测试策略
| 测试类型 | 占比 | 内容 |
|----------|------|------|
| 单元测试 | 70% | 时间转换函数、验证函数 |
| 集成测试 | 30% | 数据库约束验证、Schema初始化 |

### 测试数据
- 时间转换：正常值、进位、零值、负值（边界）
- 工作时间：边界时刻（08:30, 12:00, 13:00, 17:30）
- 数据库：有效/无效时间值、枚举约束

### 验收标准
- [x] 所有时间工具函数测试通过
- [x] 数据库约束正确拒绝负数和零值
- [x] 覆盖率 ≥ 80%

---

## 阶段2：解析引擎核心

### 目标
实现Markdown记录的解析功能，包括日期、时长、类型识别。

### 交付模块
1. **日期解析器** (`src/parsers/date_parser.py`)
   - 单日期格式（2025.08.15, 2025.9.18, 2025年8月15日）
   - 日期范围（2025.10.27-29, 2025.10.27-11.3）
   - 上下文年份继承

2. **时长解析器** (`src/parsers/hours_parser.py`)
   - 直接时长（3.5小时, 4小时）
   - 时分格式（1小时30分钟）
   - 关键词（半天=4h, 一天=8h, 三天=24h）

3. **类型识别器** (`src/parsers/type_parser.py`)
   - 加班类型（晚间/早晨/午休）
   - 请假类型（事假/病假/年假）
   - 调休类型
   - 置信度评分

### 测试策略
| 测试类型 | 占比 | 内容 |
|----------|------|------|
| 单元测试 | 60% | 各解析器独立测试 |
| 集成测试 | 40% | 解析管道组合测试 |

### 测试数据
- 日期：7种格式 × 有效/无效组合
- 时长：小数、整数、关键词、混合
- 类型：明确描述、边界描述、模糊描述

### 验收标准
- [x] 支持所有PRD定义的日期格式
- [x] 支持所有PRD定义的时长格式
- [x] 类型识别准确率 > 90%（高置信度）
- [x] 覆盖率 ≥ 80%

## 阶段3：法定节假日管理

### 目标
实现国务院节假日通知的解析和维护功能。

### 交付模块
1. **节假日通知解析器** (`src/parsers/holiday_notification_parser.py`)
   - 从文本提取节假日安排
   - 识别法定节假日 vs 调休假期
   - 识别调休上班日

2. **农历转换器** (`src/utils/lunar_converter.py`)
   - 农历日期转公历
   - 春节等农历节日计算

3. **节假日服务** (`src/services/holiday_service.py`)
   - 日期类型判断（工作日/周末/法定假日/调休）
   - 年度节假日数据管理

### 测试策略
| 测试类型 | 占比 | 内容 |
|----------|------|------|
| 单元测试 | 50% | 解析组件、转换函数 |
| 集成测试 | 40% | 完整解析流程 |
| E2E测试 | 10% | 节假日维护界面（如适用） |

### 测试数据
- 国务院通知样本（2026年完整示例）
- 农历转换对照表（腊月二十八→正月初七）
- 边界日期（调休上班日、跨月假期）

### 验收标准
- [ ] 能正确解析标准格式国务院通知
- [ ] 法定节假日识别准确率 > 95%
- [ ] 支持人工修正解析错误
- [ ] 覆盖率 ≥ 80%

## 阶段4：核心业务逻辑

### 目标
实现劳动法合规的加班统计、调休余额、工资计算。

### 交付模块
1. **加班统计服务** (`src/services/overtime_service.py`)
   - 按类型统计（工作日延时/周末/法定假日）
   - 月度/年度汇总
   - 员工排名

2. **调休余额服务** (`src/services/comp_off_service.py`)
   - FIFO抵扣算法
   - 余额计算（总获得-已使用）
   - 过期提醒

3. **工资计算服务** (`src/services/salary_service.py`)
   - 1.5倍/2倍/3倍工资计算
   - 调休抵扣后的应付工资
   - 月度工资明细

### 测试策略
| 测试类型 | 占比 | 内容 |
|----------|------|------|
| 单元测试 | 50% | 计算函数、抵扣算法 |
| 集成测试 | 50% | 服务层组合业务逻辑 |

### 测试数据
- 加班记录：各类型混合数据
- 调休场景：多笔余额、跨期抵扣
- 工资计算：不同基数、调休抵扣前后对比

### 验收标准
- [ ] 工资计算符合《劳动法》第44条
- [ ] FIFO抵扣顺序正确
- [ ] 覆盖率 ≥ 80%

## 阶段5：逐行审批与导入流程

### 目标
实现低置信度记录的人工审批和批量导入功能。

### 交付模块
1. **解析结果处理器** (`src/services/parse_result_processor.py`)
   - 置信度分级（HIGH/MEDIUM/LOW）
   - 异常记录标记

2. **存储服务** (`src/services/storage_service.py`)
   - 记录分发到对应表
   - 周末加班自动生成调休余额
   - 数据库事务管理

3. **审批会话管理** (`src/services/review_service.py`)
   - 逐条确认界面逻辑
   - 批量确认/拒绝
   - 导入报告生成

### 测试策略
| 测试类型 | 占比 | 内容 |
|----------|------|------|
| 单元测试 | 40% | 分级逻辑、存储分发 |
| 集成测试 | 50% | 完整导入流程 |
| E2E测试 | 10% | CLI交互流程 |

### 测试数据
- 置信度分级：高/中/低置信度样本
- 存储分发：各类记录到对应表
- 多时段分解：一条记录→多条存储

### 验收标准
- [ ] 低置信度记录必须人工确认
- [ ] 存储分发正确（加班→overtime_records + comp_off_balances）
- [ ] 生成完整导入报告
- [ ] 覆盖率 ≥ 80%

## 阶段依赖关系

```
阶段1 (基础) ─────────────────────────────┐
     ↓                                      │
阶段2 (解析) ───────────────────┐           │
     ↓                          ↓           │
阶段3 (节假日) ──────┐    阶段4 (业务逻辑)   │
     ↓              ↓         ↓            │
     └──────────────┴────→ 阶段5 (审批导入)  │
                              ↓            │
                           阶段6 (CLI) ←───┘
```

**关键依赖**:
- 阶段2依赖阶段1的时间工具
- 阶段4依赖阶段3的节假日数据
- 阶段5依赖阶段2的解析器和阶段4的业务逻辑

## 覆盖率要求说明

### 整体策略
- **最低覆盖率**: 80%
- **理想覆盖率**: 90%+
- **关键模块**（工资计算、调休抵扣）: 建议95%+

### 当前状态（阶段1+2完成）

| 模块 | 当前覆盖率 | 状态 |
|------|-----------|------|
| time_utils.py | 96% | ✅ 优秀 |
| schema.py | 87% | ✅ 达标 |
| date_parser.py | 81% | ✅ 达标 |
| hours_parser.py | 86% | ✅ 达标 |
| type_parser.py | 100% | ✅ 完美 |
| **平均** | **90%** | **✅** |

### 是否需要100%？

**建议**: 不需要强制100%，但应满足以下条件再进入下一阶段：

1. ✅ **所有测试通过** - 零失败
2. ✅ **核心逻辑覆盖** - 业务关键路径100%
3. ✅ **边界情况覆盖** - 错误处理、异常分支
4. ✅ **最低80%** - 整体达标

**当前评估**: 阶段1+2已满足进入阶段3的条件。
