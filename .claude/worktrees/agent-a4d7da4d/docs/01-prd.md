## 2. 项目概述

### 2.1 项目背景

目前用户使用 Markdown 文件手动记录员工的加班时长，数据格式松散、不统一。需要开发一个系统来自动化提取这些半结构化数据，进行标准化处理后存储到 SQLite 数据库，并提供查询和分析功能。

### 2.2 项目目标

1. **数据提取自动化**：自动从 Markdown 文件中提取加班记录
2. **数据标准化**：将各种格式的记录转换为统一的数据结构
3. **数据持久化**：可靠存储到 SQLite 数据库
4. **查询与分析**：提供灵活的查询和统计功能
5. **扩展性支持**：为未来支持更多记录格式预留扩展接口

### 2.3 目标用户

- 人力资源部门：需要统计员工加班情况
- 部门主管：需要查看团队加班数据
- 财务部门：需要加班费用核算数据

## 3. 功能需求

### 3.1 核心功能

#### FR-001: Markdown 文件解析
- **描述**：系统能够解析 Markdown 格式的加班记录文件
- **优先级**：P0
- **验收标准**：
  - 支持单日期格式（2025.08.15, 2025.9.18）
  - 支持日期范围格式（2025.10.27-29）
  - 自动识别并标准化日期格式

#### FR-002: 记录类型识别
- **描述**：自动识别不同类型的记录
- **优先级**：P0
- **支持的记录类型**：
  | 类型 | 示例 | 识别特征 |
  |------|------|----------|
  | 加班记录 | "晚上3.5小时"、"早7到晚10共15小时" | 包含时长描述 |
  | 请假记录 | "请假半天"、"请假三天" | 包含"请假"关键词 |
  | 调休记录 | "调休三天" | 包含"调休"关键词 |
  | 累计声明 | "累计48.5小时" | 仅作参考展示，**不用于系统计算** |

#### FR-003: 时长计算
- **描述**：从各种描述中提取准确的加班/请假时长
- **优先级**：P0
- **支持的格式**：
  - 直接时长："3.5小时"、"4.5小时"
  - 半天/全天："半天"=4小时，"一天"=8小时
  - 时间段计算："早7到晚10"自动计算为15小时
  - 范围天数："三天"=24小时，自动计算

#### FR-004: 数据存储
- **描述**：将解析后的数据存储到 SQLite 数据库
- **优先级**：P0
- **要求**：
  - 支持记录去重（基于日期+员工）
  - 支持记录更新
  - 保留原始文本用于审计

#### FR-005: 批量导入
- **描述**：支持批量导入多个 Markdown 文件
- **优先级**：P1
- **要求**：
  - 支持目录遍历
  - 提供导入进度反馈
  - 生成导入报告（成功/失败统计）

#### FR-006: 数据查询
- **描述**：提供灵活的数据查询功能
- **优先级**：P1
- **查询维度**：
  - 按员工查询
  - 按日期范围查询
  - 按记录类型查询
  - 按部门查询（未来扩展）

#### FR-007: 统计分析
- **描述**：提供加班数据统计功能
- **优先级**：P1
- **统计维度**：
  - 月度加班统计
  - 年度加班统计
  - 员工加班排名
  - 合规调休余额追踪（按劳动法规则独立计算）

> **注意**：系统中所有余额计算必须根据《劳动法》规则独立进行，**不可参考**员工记录文件中声明的"累计"值。因为员工记录方式未区分加班类型（工作日延时/周末/法定假日），其累加方式不符合合规要求。

#### FR-008: 加班类型自动识别（劳动法合规）
- **描述**：自动识别加班类型，区分工作日延时、周末、法定节假日
- **优先级**：P0
- **工作时间定义**：
  - 工作日：周一到周五
  - 上午：08:30 - 12:00（3.5小时）
  - 下午：13:00 - 17:30（4.5小时）
  - 标准工作时长：8小时/天
- **验收标准**：
  - 根据日期自动判断是工作日、周末还是法定节假日
  - 支持配置法定节假日列表
  - 工作日延时加班：
    - 早晨加班：08:30 之前
    - 午休加班：12:00 - 13:00
    - 晚间加班：17:30 之后
  - 周末加班：周六、周日全天
  - 法定节假日加班：按国家假日办发布的节假日安排

#### FR-009: 合规工资计算
- **描述**：根据劳动法规定计算应发工资和可调休时长
- **优先级**：P0
- **计算规则**：
  | 加班类型 | 工资倍数 | 可调休 |
  |----------|----------|--------|
  | 工作日延时 | 1.5倍 | ❌ 不可调休，必须支付工资 |
  | 周末 | 2倍 | ✅ 可调休，不调休则支付2倍工资 |
  | 法定节假日 | 3倍 | ❌ 不可调休，必须支付3倍工资 |
- **输出**：
  - 每月应发加班工资明细
  - 可调休时长余额
  - 不可调休的加班工资（按1.5倍/3倍计算）

#### FR-010: 调休余额管理
- **描述**：实时计算和管理员工可调休时长
- **优先级**：P1
- **功能要求**：
  - 仅周末加班可转为调休时长
  - 调休时长有效期管理（建议6个月内使用）
  - 调休使用时优先抵扣最早到期的时长
  - 实时显示当前可用调休余额
  - 调休申请与审批流程（可选）

  - 支持人工修正解析错误
  - 数据保存前需用户确认

## 4. 非功能需求

### 4.1 性能需求

| 指标 | 目标值 | 说明 |
|------|--------|------|
| 单文件解析时间 | < 1秒 | 1000行以内的文件 |
| 批量导入速度 | > 100条/秒 | 批量处理性能 |
| 查询响应时间 | < 500ms | 单条件查询 |
| 数据库容量 | > 100万条 | 单表记录数 |

### 4.2 可靠性需求

- **数据完整性**：解析失败时保留原始数据，不丢失信息
- **错误恢复**：支持断点续传，批量导入可从中断处继续
- **事务支持**：数据库操作支持事务，保证数据一致性

### 4.3 可维护性需求

- **日志记录**：完整的操作日志和错误日志
- **配置化**：解析规则可配置，便于适应新格式
- **模块化**：各组件职责清晰，便于维护和扩展

### 4.4 安全性需求

- **数据隔离**：不同部门数据逻辑隔离
- **访问控制**：操作权限控制（未来版本）
- **审计追踪**：记录数据变更历史

### 4.5 劳动法合规性需求

#### 4.5.1 加班类型判定准确性
- **要求**：系统必须准确识别加班发生的日期类型
- **验证**：与国家发布的法定节假日安排对比验证
- **容错**：对边界情况（调休上班日）提供人工确认机制

#### 4.5.2 工资计算合规性
- **要求**：严格按《劳动法》第44条规定计算
  - 工作日延时：支付不低于工资的150%
  - 休息日工作：支付不低于工资的200% 或 安排补休
  - 法定假日工作：支付不低于工资的300%（不得补休替代）
- **审计**：保留计算过程日志，支持追溯

#### 4.5.3 调休管理合规性
- **要求**：仅允许将周末加班转为调休时长
- **限制**：工作日延时和法定节假日加班不得调休
- **有效期**：调休时长建议6个月内使用，逾期提醒
- **抵扣顺序**：先到期先抵扣（FIFO）

### 4.5 兼容性需求

- **编码支持**：支持 UTF-8 编码的 Markdown 文件
- **换行符**：支持 LF 和 CRLF 换行符
- **日期格式**：兼容多种日期分隔符（.、/、-）

## 5. 数据示例

### 5.1 输入数据示例

#### 基本记录示例
```markdown
2025.08.15，下午请假回老家，请假半天。
2025.09.10，晚上3.5小时
2025.9.18，晚上4.5小时
2025.10.25，早7到晚10共15小时
2025.10.27-29，请假三天，减24小时
2025.12.29-31，调休三天
```

#### 多时段加班示例（工作日）
```markdown
2025.10.24，早晨1.5小时，晚上5.5小时
2025.11.05，中午12:30-13:00加班，晚上6小时
2025.11.06，早7点到岗，晚上加班到20:00
```

#### 说明
- **工作日延时加班**可能在以下时段：
  - 早晨：08:30 之前（如"早7点到岗"）
  - 午休：12:00 - 13:00（如"中午12:30-13:00加班"）
  - 晚间：17:30 之后（如"晚上3.5小时"）
- **周末加班**：周六、周日全天工作（如"早7到晚10"）
- **调休**：使用周末加班产生的可调休余额

## 6. 约束条件

### 6.1 技术约束

- 使用 Python 作为主要开发语言
- 使用 SQLite 作为数据库（便于部署和迁移）
- 不依赖外部服务，支持离线运行

### 6.2 业务约束

- ~~累计余额为实时计算值，系统需要能够验证计算正确性~~ ⚠️ **修订**：系统中所有余额计算必须**独立进行**，**不可参考**员工记录文件中的"累计"声明值。因为员工记录未区分加班类型（工作日延时/周末/法定假日），其累加方式不符合《劳动法》合规要求。
- 请假和调休需要区分处理（可能影响薪资计算规则）
- 日期范围记录需要展开为单日记录或特殊标记

### 6.3 时间约束

- 第一阶段：核心功能开发（4周）
- 第二阶段：测试与优化（2周）
- 第三阶段：部署与文档（1周）
